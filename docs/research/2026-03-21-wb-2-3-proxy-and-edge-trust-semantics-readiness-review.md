# Scope reviewed

`WB-2.3` is the next verified-identity tranche after bounded native directory discovery. Its job is to make the proxy and edge trust contract explicit for signature-relevant headers without changing authorization policy, dashboard/operator surfaces, or the core native/provider verification contracts.

# Findings

1. The native verifier in `src/bot_identity/native_http_message_signatures.rs` currently consumes:
   - `Signature`,
   - `Signature-Input`,
   - `Signature-Agent`,
   - and derived request components such as `@scheme`, `@authority`, `@target-uri`, and `@path`.
2. The derived HTTPS context already follows an explicit trust model:
   - shared-host/gateway deployments may treat `X-Forwarded-Proto` or `Forwarded: proto=...` as HTTPS only when `SHUMA_FORWARDED_IP_SECRET` matches `X-Shuma-Forwarded-Secret`,
   - edge deployments may treat `spin-full-url=https://...` as HTTPS when `gateway_deployment_profile=edge-fermyon`.
3. Provider-normalized edge assertions in `src/providers/external.rs` already require the forwarded-secret trust gate before `x-shuma-edge-verified-identity-*` headers can verify. That existing fail-closed posture is the contract to preserve, not broaden.
4. The gateway forwarder in `src/runtime/upstream_proxy.rs` already strips:
   - hop-by-hop headers,
   - forwarded-provenance headers,
   - and all `x-shuma-*` request headers.
   That means Shuma-owned trust headers already do not pass through the gateway to the upstream origin.
5. The gateway forwarder does not explicitly document or test the handling of native signature headers. Today they are forwarded only because they are ordinary client request headers, not because the contract is frozen anywhere.
6. The gateway request rewrite changes signature-relevant context for any upstream application that tries to re-verify the original request after the proxy hop:
   - `Host` is not forwarded from the client,
   - `Forwarded` and `X-Forwarded-*` are regenerated by Shuma,
   - the target URI authority is rewritten to the configured upstream origin.
   Therefore Shuma must treat native verification as a local trust decision taken before proxy mutation, and the docs must state that clearly.
7. `WB-2.3` should stay narrow. The following remain out of scope:
   - named identity authorization policy and low-cost profiles (`WB-3.*`),
   - trusted-directory UI/admin surfaces,
   - new outbound directory behavior,
   - or any new global connection/auth/polling/dashboard state changes.

# Decisions

1. `Signature`, `Signature-Input`, and `Signature-Agent` remain unprivileged client-supplied inputs. They must not be stripped by Shuma merely because they participate in native verification, and they must not be trusted without valid signature verification.
2. Shuma-owned trust headers remain privileged and proxy-owned:
   - `X-Shuma-Forwarded-Secret`,
   - `X-Shuma-Edge-Verified-Identity-*`,
   - and any other `x-shuma-*` headers.
   They must continue to be stripped on gateway forward and must never be treated as trustworthy without the existing forwarded-secret gate.
3. Native verification must continue to fail closed when HTTPS-derived signature components rely on untrusted forwarded proto headers. A request signed as `https` must not verify through shared-host forwarding unless the forwarded-secret gate validates.
4. `WB-2.3` should add focused regression coverage for both trust surfaces:
   - trusted forwarding and edge HTTPS context for the native verifier,
   - gateway pass-through of signature headers alongside stripping of Shuma-owned assertion headers.
5. Operator-facing docs must make the mutation risk explicit: upstream applications behind the Shuma gateway must not assume the original authority or Shuma-owned assertion headers survive the proxy hop, so any upstream re-verification must be designed with that rewritten request shape in mind.

# Why this is the right next slice

1. It closes the remaining trust-boundary gap between the now-working native verifier and the gateway/edge deployment modes Shuma already supports.
2. It keeps the verified-identity roadmap honest by freezing trust semantics before any authorization policy starts depending on them.
3. It preserves the repo's existing secure defaults: provider assertions stay gated, forwarded HTTPS context stays explicit, and the gateway does not become a silent carrier of privileged Shuma trust headers.
